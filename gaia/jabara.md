# Rewriting the Jabara emitter for Gaia

The current emitter writes a restricted x86-64 assembly dialect. Gaia replaces
that dialect as the compiler's canonical output. This is a source-language
change in `jabara/src/jc/emitter.jabara`; Jabara itself remains the implementation
language of the compiler.

## Observed compiler requirements

An audit of every emission path in the current emitter found these semantic
requirements:

- load constants and symbol addresses;
- load and store 64-bit values;
- zero-extend byte loads and write byte stores;
- stack push and pop;
- frame entry and callee-cleaning return;
- direct and indirect call;
- unconditional and Boolean conditional branch;
- add, subtract, low multiply, unsigned divide, unsigned remainder;
- bitwise AND, OR, XOR, complement, and two's-complement negation;
- logical left and right shifts;
- signed equality and ordering comparisons producing `0` or `1`;
- labels, globals, alignment, strings, and zero-initialized data.

No Jabara emission path requires floating point, SIMD, atomics, port I/O,
privileged instructions, x86 string operations, or a general flags register.

## Register rewrite

The existing compiler conventions map directly:

| Current x86 name | Gaia name |
| --- | --- |
| `rax` / `eax` / `al` | `a` |
| `rdi` / `edi` / `dil` | `b` |
| `rcx` / `cl` | `c` |
| `rdx` / `edx` / `dl` | `d` |
| `rbp` | `fp` |
| `rsp` | `sp` |
| implicit RIP | `ip` |
| zeroing idiom | `z` |

Jabara never needs the other x86 general registers in generated code.

## Pattern rewrite

| Current output | Gaia output |
| --- | --- |
| `mov rax, value` | `mov a, value` |
| `lea rax, [rel label]` | `addr a, label` |
| `mov rax, [base + off]` | `ld a, [base + off]` |
| `movzx eax, byte [base + off]` | `ldb a, [base + off]` |
| `mov [base + off], rax` | `st [base + off], a` |
| `mov byte [base + off], al` | `stb [base + off], a` |
| `xor edx, edx` | `mov d, z` |
| `add rax, rdi` | `add a, a, b` |
| `neg rax` | `sub a, z, a` |
| `not rax` | `xor a, a, -1` |
| `cmp` + `setcc` + `movzx` | one Gaia comparison |
| `test rax, rax` + `jz L` | `bz a, L` |
| `cmp rax, rdi` + inverse branch | comparison into a temporary, then `bz`/`bnz` |
| raw `ENTER` bytes | `enter size` |
| raw `LEAVE`, `RET`, `RET n` bytes | `ret n` |

The emitter should prefer direct Boolean branches. For example, instead of
materializing a comparison and then testing it when no expression value is
needed:

```text
lt  b, a, limit
bz  b, done
```

A later optimizer may fuse that pair in native encoding, but the source remains
flagless and explicit.

## Three-operand opportunities

Gaia eliminates several x86 shuffles:

Current:

```text
push rax
... evaluate right into rax ...
pop rdi
add rax, rdi
```

Initial Gaia can preserve the same evaluation strategy:

```text
push a
... evaluate right into a ...
pop b
add a, a, b
```

After correctness is established, the emitter can exploit three operands and
choose evaluation order more freely. That optimization is separate from the
first migration.

## Direct frame output

The current compiler injects raw bytes for `push [rbp + disp8]` and for
`enter`/`leave`/`ret n` because those x86 forms were missing or inconvenient in
Orgasm. Gaia source must contain no such byte-level instruction escapes.

Direct frame values become ordinary operations:

```text
ld   a, [fp + offset]
push a
```

Native Gaia may eventually fuse a load-and-push sequence if measurement justifies
it. Source correctness must not depend on the fusion.

## Compiler output header

The current generated header:

```text
; generated by the jabara compiler
bits 64
```

becomes:

```text
; generated by the jabara compiler
gaia 1
```

`gaia 1` selects the source-language version. It is an assembler directive, not
an instruction.

## Required compiler tests

The migration is complete only when the Gaia emitter covers and tests:

- every unary and binary operator;
- constant and variable shifts;
- all six comparisons as values and branches;
- direct calls with zero, one, and many arguments;
- indirect closure calls;
- nested closures and environment traversal;
- globals, locals, parameters, records, members, dereference, and byte access;
- strings and data labels;
- large and negative immediates;
- frame sizes and argument cleanup at their current limits.

The same generated Gaia assembly must build through both Orgasm and the native
Gaia assembler.
